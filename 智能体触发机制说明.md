# 智能体触发机制说明（基于当前代码实现）

本文档描述当前项目中“智能体自动触发”的真实实现逻辑，覆盖：

1. 触发条件总览与判定方法
2. 按每个智能体拆解触发机制

---

## 一、当前有哪些触发条件？如何判定？

当前后端的触发入口主要有三类：

1. `@提及触发（mention）`
2. `沉默超时触发（silence）`
3. `垄断发言触发（monopoly）`
4. `委员会定时分析触发（committee timer）`

> 说明：虽然有 4 类机制，但是否真正生效还受 `agent_settings.yaml` 的开关控制。

### 1) @提及触发（mention）

- 触发入口：学生通过 WebSocket 发送 `chat:message`，携带 `mentions` 字段。
- 判定方式：
1. 对 `mentions` 做标准化：去空格、转小写、去重。
2. 按 `mention.max_mentions_per_message` 截断（至少允许 1 个）。
3. 若角色存在于 `ROLE_AGENTS`，则入队任务；不存在则回发 `unsupported`。
- 任务特征：
1. `trigger_type = "mention"`
2. 优先级来自配置 `mention.priority`（当前为 `0`，最高优先级）
3. 会记录 `source_message_id`、`student_name` 供后续回复上下文使用

### 2) 沉默超时触发（silence）

- 触发入口：调度器周期任务 `check_silence`（每 30 秒运行一次）。
- 判定方式（全部满足才触发）：
1. 房间在 `active_rooms` 集合中（即被标记为活跃房间）
2. 已有 `last_msg_time`
3. 房间已过预热期 `warmup_minutes`（避免刚开房就触发）
4. `当前时间 - last_msg_time >= silence_threshold_seconds`
5. 不存在 `trigger_lock:{room_id}:silence` 防抖锁
- 命中后行为：
1. 写入防抖锁（TTL = `silence_threshold_seconds + 60`）
2. 入队 `facilitator` 任务，`trigger_type = "silence"`，`priority = 2`

### 3) 垄断发言触发（monopoly）

- 触发入口：每条学生消息落库后，立即执行 `check_monopoly`。
- 判定方式（核心是“最近 N 条消息是否都来自同一人”）：
1. 把当前发送者 `sender_id` 压入 Redis 列表 `room:{room_id}:recent_senders`
2. 列表裁剪到最近 `monopoly_message_count` 条（默认 5）
3. 若列表长度不足阈值，不触发
4. 若这 N 条发送者去重后数量不为 1（不全是同一人），不触发
5. 若存在 `trigger_lock:{room_id}:monopoly`（60 秒防抖锁），不触发
- 命中后行为：
1. 写入 60 秒锁，避免连续重复触发
2. 入队 `encourager` 任务，`trigger_type = "monopoly"`，`priority = 2`

### 4) 委员会定时分析触发（committee timer）

- 触发入口：调度器周期任务 `check_committee_timer`（按 `analysis_interval_minutes`，当前 5 分钟）。
- 判定方式：
1. 遍历 `active_rooms`
2. 拉取最近消息和成员信息
3. 通过 `BasicCommittee` 做三类分析：认知、情绪、互动
4. 根据分析结果调用 `_dispatch` 生成干预任务
- `_dispatch` 规则：
1. `devil_advocate`：观点多样性 `diversity_score` 低于阈值
2. `summarizer`：检测到冲突表达
3. `encourager`：检测到消极表达且参与度失衡
- 任务特征：
1. `trigger_type = "committee"`
2. `priority = 1`（高于 silence/monopoly，低于 mention 的 0）

## 二、按每个智能体拆解触发机制

> 角色集合：`facilitator`、`devil_advocate`、`summarizer`、`resource_finder`、`encourager`

### 1) Facilitator（主持人）

包含的触发条件：

1. `mention`（被 `@facilitator`）
2. `silence`（房间沉默超时）

触发逻辑细节：

1. `mention` 命中后直接入队，优先级通常最高（默认 0）。
2. `silence` 由调度器轮询判定，满足“活跃房间 + 超过预热 + 沉默超时 + 无防抖锁”才入队。
3. 进入 Worker 后还会经过二次门控：是否启用 `silence_trigger_enabled` 与 `facilitator_silence_enabled`。
4. 真正执行时，角色 Agent 生成流式消息并广播给房间。

### 2) Devil Advocate（批判者）

包含的触发条件：

1. `mention`（被 `@devil_advocate`）
2. `committee`（委员会判定“观点多样性不足”）

触发逻辑细节：

1. `mention`：用户主动召唤即入队。
2. `committee`：当 `diversity_score < diversity_score_threshold` 时，委员会产生该角色任务，优先级 1。
3. Worker 执行前会检查 `committee_enabled` 及 `committee_devil_advocate_enabled`。

### 3) Summarizer（总结者）

包含的触发条件：

1. `mention`（被 `@summarizer`）
2. `committee`（委员会判定出现冲突表达）

触发逻辑细节：

1. `mention`：学生主动召唤即入队。
2. `committee`：当情绪分析命中冲突关键词时，生成 `summarizer` 干预任务（优先级 1）。
3. Worker 执行前会检查 `committee_enabled` 及 `committee_summarizer_enabled`。

### 4) Resource Finder（资源检索者）

包含的触发条件：

1. `mention`（被 `@resource_finder`）

触发逻辑细节：

1. 当前实现中没有独立规则触发，也不在委员会 `_dispatch` 输出列表中。
2. 因此其自动触发来源仅为用户显式 `@`。
3. 命中后直接进入队列，按 mention 优先级执行。

### 5) Encourager（鼓励者）

包含的触发条件：

1. `mention`（被 `@encourager`）
2. `monopoly`（检测到垄断发言）
3. `committee`（消极表达 + 参与失衡）

触发逻辑细节：

1. `mention`：学生主动召唤即入队。
2. `monopoly`：每条学生消息后实时检查“最近 N 条是否同一人”；命中后带 60 秒防抖锁入队。
3. `committee`：委员会同时命中“passive=true 且 balance_score 低于阈值”时入队。
4. Worker 执行前会分别检查 `monopoly_encourager_enabled` 和 `committee_encourager_enabled`。

## 三、当前配置下的生效状态（非常关键）

当前 `backend/app/agents/agent_settings.yaml` 中：

1. `mention.enabled: true`（开启）
2. `timing.silence_trigger_enabled: false`（关闭）
3. `auto_speak.facilitator_silence_enabled: false`（关闭）
4. `auto_speak.monopoly_encourager_enabled: false`（关闭）
5. `auto_speak.committee_enabled: false`（关闭）
6. `committee_*_enabled` 也均为 `false`

因此在“当前默认配置”下，实际可稳定触发的主通道是：

1. `@提及触发（mention）`

而 `silence`、`monopoly`、`committee` 相关自动发言，即使代码存在，也会被配置门控拦截。

## 四、补充：统一执行门控（所有角色都会经过）

任何触发任务进入队列后，在真正执行前都会被 Worker 做统一控制：

1. 功能开关检查（按 `trigger_type` 与角色对应开关）
2. 每房间每小时干预上限（`global_intervention_limit_per_hour`）
3. 角色冷却（`cooldown:{room_id}:{agent_role}`，默认 60 秒）
4. 房间互斥锁（`room:{room_id}:agent_lock`，同一时刻仅一个 Agent 发言）
5. 若冷却或锁冲突，会延迟重入队列，而非直接丢弃

这部分机制保证了：即便触发条件频繁命中，也不会出现多个智能体同一时刻“抢话筒”。
